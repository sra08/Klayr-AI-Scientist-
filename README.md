# Klayr

**AI Co-Scientist Platform**

Klayr is a production-grade AI research platform that automates the full lifecycle of scientific experiment planning — from raw hypothesis to detailed, reviewer-ready research plans. It combines a stateful multi-agent pipeline, live literature retrieval, semantic novelty detection, and a feedback-driven learning loop designed for continuous improvement in real lab environments.

## Overview

Scientific research planning is time-intensive, fragmented across tools, and difficult to standardize. Klayr addresses this by providing an end-to-end AI platform where researchers submit a hypothesis and receive a fully structured experiment plan — including literature review, protocol, materials list, budget, and timeline — generated by a coordinated team of domain-specific AI agents.

The platform is built around the principle that AI assistance in science should be **correctable and improving**. Every feedback signal from a researcher is persisted and used to refine future outputs, making the system measurably more accurate over time within a given research domain.

## Architecture

Klayr follows a clean hexagonal (ports and adapters) architecture separating domain logic from infrastructure concerns. The AI pipeline is implemented as a directed acyclic graph using LangGraph, where each node is an independent agent responsible for a single research planning stage.

```
Client (React/Vite)
        |
        v
FastAPI REST API  (src/api/)
        |
        v
Domain Layer      (src/domain/)
  ├── Pipeline Graph  (LangGraph stateful DAG)
  ├── Prompts         (per-agent prompt builders)
  ├── Parsers         (structured output extraction)
  └── Ports           (abstract interfaces)
        |
        v
Infrastructure    (src/infrastructure/)
  ├── PostgreSQL      (experiment & researcher persistence)
  ├── Redis           (caching layer)
  ├── Groq            (LLM inference — Llama 3.3 70B)
  └── ArXiv / PubMed  (live literature retrieval)
```

### Pipeline Execution Flow

A single pipeline run passes through the following agent nodes in sequence:

| Step | Agent | Output |
|------|-------|--------|
| 1 | Hypothesis Refiner | Structured, falsifiable hypothesis |
| 2 | Literature QC Agent | Novelty assessment with cited sources |
| 3 | Protocol Architect | Step-by-step experimental protocol |
| 4 | Resource Specialist | Materials list with verified supplier references |
| 5 | Budget Analyst | Itemized project budget with optimization notes |
| 6 | Timeline Manager | Phase breakdown with dependency mapping |
| 7 | Validation Agent | Internal consistency check across all sections |

## Features

### Multi-Agent Research Pipeline

Each agent operates on a shared `PipelineState` object, receiving prior context and contributing a discrete, structured output. Agents are independently testable and can be updated without affecting the rest of the graph.

### Feedback-Driven Learning Loop

Researchers can submit structured corrections on any section of a generated plan. These corrections are stored as domain-scoped "Prior Learnings" and are automatically prepended to relevant agent prompts on future runs. The system does not require retraining — improvements are prompt-level and take effect immediately.

### Live Novelty Detection

Before generation, Klayr performs a semantic similarity search against its internal experiment database. Hypotheses are classified as:

| Status | Meaning |
|--------|---------|
| Original | No semantically similar work exists in the system |
| In Progress | A researcher in the network has an active plan on this topic |
| Completed | A finalized experiment plan already exists for this hypothesis |

### Literature Integration

The platform queries ArXiv and PubMed in parallel, retrieves up to 30 sources per run, and generates plain-language summaries using the active LLM. Results are ranked by cosine similarity to the input hypothesis using `sentence-transformers`.

### Collaboration Network

Researchers can discover collaborators based on domain overlap and hypothesis similarity. The platform maintains a graph of researcher profiles and active projects to facilitate matching.

## Getting Started

### Prerequisites

| Requirement | Minimum Version |
|-------------|-----------------|
| Python | 3.10 |
| Node.js | 18.0 |
| PostgreSQL | 14 (or Neon serverless) |
| Redis | 6.0 |

### 1. Clone the Repository

```bash
git clone <repository-url>
cd Klayr-AI-Scientist-
```

### 2. Configure Environment Variables

```bash
cp .env.example .env
# Edit .env with your credentials (see Environment Variables section)
```

### 3. Initialize the Database

```bash
python scripts/init_db.py
```

### 4. Start the Backend

```bash
pip install -r requirements.txt
python -m src.api.main
```

The API will be available at `http://localhost:8000`. Interactive API documentation is served at `http://localhost:8000/docs`.

### 5. Start the Frontend

```bash
cd frontend/frontend
npm install
npm run dev
```

The development server will be available at `http://localhost:5173`.

## Environment Variables

Create a `.env` file in the project root. All variables listed below are required unless marked optional.

| Variable | Description |
|----------|-------------|
| `DATABASE_URL` | PostgreSQL connection string (e.g., `postgresql+asyncpg://user:pass@host/db`) |
| `GROQ_API_KEY` | API key for Groq inference |
| `REDIS_URL` | Redis connection string (e.g., `redis://localhost:6379`) |

## API Reference

The FastAPI backend exposes a versioned REST API. Full interactive documentation is available at `/docs` (Swagger UI) and `/redoc` (ReDoc) when the server is running.

| Prefix | Tag | Description |
|--------|-----|-------------|
| `/api` | `pipeline` | Submit a hypothesis and run the generation pipeline |
| `/api` | `plans` | Retrieve, list, and manage generated experiment plans |
| `/api` | `feedback` | Submit and retrieve researcher feedback on plan sections |
| `/api` | `profile` | Researcher profile management and authentication |
| `/api/researchers` | `researchers` | Researcher search and directory |
| `/api/collab` | `collaboration` | Collaboration requests and matching |
| `/api/notifications` | `notifications` | In-platform notification management |

3. Ensure all new agents implement the port interfaces defined in `src/domain/ports/`.
4. Submit a pull request with a clear description of the change and its motivation.
